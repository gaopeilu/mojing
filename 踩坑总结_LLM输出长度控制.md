# LLM 应用踩坑实录：输出控制与性能优化

## 📅 背景信息

**项目**：墨镜 (MoJing) 智能写作分身（FastAPI + LangGraph + DashScope）  
**时间**：2026-05-17  
**核心问题**：
1. 生成大纲时，模型持续输出 1000+ 字，无视提示词约束
2. 反思节点响应过慢，用户体验差

---

## 🎯 问题一：LLM 输出长度失控

### 初始需求
- 大纲字数：**严格控制在 100 字以内**
- 正文字数：**控制在 800 字左右（700-900字）**

### 遇到的问题
```python
# 提示词中明确要求
prompt = """
# 要求：
1. 大纲严格控制在100字以内
2. 逻辑清晰，使用 Markdown 标题层级
...
"""

# 但实际输出
outline_text = "......"  # 1000+ 字 😱
```

即使多次重试，模型依然输出超长内容，完全忽略提示词中的字数约束。

---

## 🔍 问题分析

### 第一次尝试：强化提示词 ❌

```python
prompt = """
# 【重要要求】
1. **大纲必须严格控制在100字以内**（包括标点符号）
2. 禁止展开详细说明，禁止举例
3. 如果超过100字，请自动删减到100字以内
"""
```

**结果**：无效。模型仍然输出 800-1000 字。

**原因分析**：
- 提示词对 LLM 来说只是"建议"，不是硬性约束
- `qwen-math-turbo` 模型擅长数学推理，但不擅长遵循简洁性指令
- 模型倾向于"填满可用空间"，给出更详细的答案

---

### 第二次尝试：添加 max_tokens 限制 ⚠️ 部分有效

```python
outline_text = call_dashscope_llm(prompt, model=outline_model, max_tokens=300)
```

**结果**：有所改善，但仍然输出 200-300 字（约 150-200 字符）。

**原因分析**：
- `max_tokens=300` 是硬上限，但模型会尽量填满这 300 tokens
- Token 与中文字符的对应关系：1 token ≈ 0.5-0.7 中文字符
- 300 tokens ≈ 150-200 中文字符，仍然超出 100 字要求

---

### 第三次尝试：后端强制截断 ❌ 破坏语义

```python
if len(clean_outline) > 100:
    outline_text = outline_text[:100] + "..."
```

**结果**：字数达标，但语义被破坏。

**用户反馈**：
> "不要截断，截断会破坏语义"

**比喻**：就像说话说到一半突然被掐断，很不自然。

例如：
```
原文：人工智能的发展经历了三个阶段：首先是符号主义AI，然后是连接主义...
截断：人工智能的发展经历了三个阶段：首先是符号主义AI，然后是连...
```
→ 句子不完整，读者无法理解。

---

## ✅ 最终解决方案（三重保障）

### 方案核心思路

**不依赖单一手段，而是组合多种技术**：

1. **更换更听话的模型**（从根源解决）
2. **降低 max_tokens**（硬上限控制）
3. **优化提示词的具体性**（软引导）

---

### 实施步骤

#### 1️⃣ 更换模型

```python
# configs/config.py

# 之前
outline_model = "qwen-math-turbo"  # 擅长数学，但话痨

# 之后
outline_model = "qwen-turbo"       # 更听话，适合简单任务
```

**为什么有效**：
- `qwen-math-turbo`：针对数学推理优化，倾向于详细解释
- `qwen-turbo`：通用模型，更容易遵循简洁性指令

---

#### 2️⃣ 降低 max_tokens

```python
# agent/workflow.py

# 之前
outline_text = call_dashscope_llm(prompt, model=outline_model, max_tokens=300)

# 之后
outline_text = call_dashscope_llm(prompt, model=outline_model, max_tokens=200)
```

**Token 与字数对照表**：

| max_tokens | 约等于中文字数 | 适用场景 |
|-----------|--------------|---------|
| 100 | 50-70 字 | 太短，可能不完整 |
| **200** | **100-140 字** | **刚好适合大纲** ✅ |
| 300 | 150-200 字 | 有点长 |
| 4000 | 2000-2800 字 | 适合正文 |

---

#### 3️⃣ 优化提示词（关键改进）

**之前的提示词**（抽象）：
```python
prompt = """
# 要求：
1. 大纲严格控制在100字以内
2. 逻辑清晰，使用 Markdown 标题层级
"""
```

**之后的提示词**（具体）：
```python
prompt = """
# 【重要要求】
1. **大纲必须极其简洁，控制在100字左右**（包括参考资料说明）
2. 使用 Markdown 标题层级（# ## ###）
3. 只保留核心框架，不要展开说明
4. 不要举例，不要详细解释
5. 每个要点用简短的短语表达（不超过15字）
6. 末尾附上参考资料说明
"""
```

**关键改进点**：

| 改进前 | 改进后 | 效果 |
|-------|-------|------|
| "控制在100字以内" | "每个要点用简短的短语表达（不超过15字）" | 给模型具体的操作方法 |
| "逻辑清晰" | "只保留核心框架，不要展开说明" | 明确什么不该做 |
| （无） | "不要举例，不要详细解释" | 禁止冗余内容 |

**比喻**：
- ❌ 抽象指令："做少一点" → 厨师不知道什么叫"少"
- ✅ 具体指令："每道菜不超过3个食材" → 厨师知道怎么做

| 方案 | 字数范围 | 语义完整性 | 稳定性 |
|------|---------|-----------|--------|

## 💡 核心技术洞察

### 1. LLM 的本质特性

**LLM 不是确定性程序，而是概率模型**：
- 提示词只是影响输出的因素之一
- 模型有自身的"偏好"和"倾向"
- 不同模型对同一提示词的响应差异巨大

**比喻**：
- 传统程序：输入 A → 必然输出 B（确定性）
- LLM：输入 A → 可能输出 B/C/D（概率性）

---

### 2. 控制 LLM 输出的三层策略

```
┌─────────────────────────────────┐
│  第1层：模型选择（最重要）        │  ← 选对模型，事半功倍
├─────────────────────────────────┤
│  第2层：参数限制（硬约束）        │  ← max_tokens、temperature
├─────────────────────────────────┤
│  第3层：提示词工程（软引导）      │  ← 具体、明确的指令
└─────────────────────────────────┘
```

**优先级**：模型选择 > 参数限制 > 提示词优化

---

### 3. 防御性编程思维

**不要完全信任外部系统**（包括 AI 模型）：

```python
# 错误思维：相信模型会遵守提示词
outline = llm.generate(prompt)  # 假设一定≤100字

# 正确思维：多层验证 + 降级处理
outline = llm.generate(prompt, max_tokens=200)
if len(outline) > 150:
    logger.warning("大纲偏长，建议用户手动精简")
    # 或者提供"精简"按钮让用户二次处理
```

## 📝 经验总结

### ✅ 最佳实践

1. **选择合适的模型**：不同模型有不同的"性格"，根据任务特点选择
2. **设置硬上限**：max_tokens 是最后的防线
3. **提示词要具体**：避免抽象指令，给出可操作的建议
4. **监控和日志**：记录实际输出，便于分析和优化
5. **接受一定的灵活性**：LLM 是概率模型，允许合理的波动范围

### ❌ 常见误区

1. **过度依赖提示词**：提示词不是万能的
2. **强制截断**：破坏语义，用户体验差
3. **追求绝对精确**：LLM 本质是概率模型，应接受合理误差
4. **忽视模型特性**：不同模型适合不同任务
